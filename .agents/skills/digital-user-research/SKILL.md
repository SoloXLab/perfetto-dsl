# Digital User Research Skill

基于公开互联网数据，完成一套完整的被动式用户研究，最终输出**用户旅程地图 + 设计建议报告**。

## 配套资产

- Workflow：`/workspace/.agents/workflows/digital-user-research/WORKFLOW.md`
- Issue 模板：`/workspace/.agents/workflows/digital-user-research/assets/issue-digital-user-research.md`

---

## 适用场景

- 无法做一手用户访谈
- 想快速了解某款 App / 产品的用户真实反馈
- 需要从大量评论中提炼结构化洞察
- 想用用户旅程视角发现产品体验断点

---

## 数据来源支持

| 来源类型 | 获取方式 |
|---|---|
| 用户直接粘贴的文本 | 直接分析 |
| App Store / Google Play 评论 | 使用 web_search + web_fetch 抓取，或调用 gplay-review-fetcher skill |
| 小红书 / 微博 / 知乎 / Reddit | 使用 web_search 搜索相关讨论 |
| 其他社区论坛 | 使用 web_fetch 获取页面内容 |

**数据收集原则：**
- 优先使用用户已提供的数据
- 若用户只提供了 App 名称或研究主题，主动用 web_search 收集数据
- 尽量收集 **正面评价（好评）、负面评价（差评）、中性分享** 三类，确保全面性
- 数据量建议：至少 30 条评论才能做有效分析，不足时告知用户

## 执行流程

### Stage 0：明确研究目标

在开始前，确认以下信息（若用户已提供则跳过）：
- **研究对象**：哪款产品 / 哪个功能模块？
- **研究焦点**：是整体体验，还是某个具体场景（如“消息通知”“支付流程”）？
- **数据来源**：用户是否已提供评论？还是需要自行收集？

### Stage 1：数据收集与预处理

1. 汇总所有评论数据。
2. 按情感倾向初步分类：正面 / 负面 / 中性。
3. 去除无效数据：广告、无意义内容、重复内容。
4. 统计基本数据：评论总数、各情感占比、数据来源分布。

输出示例：

```text
数据概览：共收集 XX 条评论
- 负面评价：XX 条（XX%）
- 正面评价：XX 条（XX%）
- 中性分享：XX 条（XX%）
来源：App Store XX 条 / 知乎讨论 XX 条 / ...
```

### Stage 2：主题聚类分析

将评论按**话题**归类，识别高频问题和诉求。

**聚类维度：**
- 功能问题（某功能不好用 / 缺失）
- 性能问题（卡顿、崩溃、耗电）
- 体验问题（操作繁琐、找不到入口）
- 情感诉求（焦虑、失控感、信任感）
- 正向反馈（哪些做得好）

每个主题需包含：
- 主题名称
- 代表性原始评论（1-3 条，直接引用用户原话）
- 出现频次 / 占比
- 情感强度（强烈 / 一般）

### Stage 3：用户旅程还原

根据数据量，自动选择旅程粒度：

**数据量 >= 50 条 -> 宏观旅程**（适合整体产品分析）

```text
发现产品 -> 注册/登录 -> 核心功能使用 -> 遇到问题 -> 解决/放弃 -> 留存/流失
```

**数据量 < 50 条，或有明确研究焦点 -> 功能级旅程**（适合某一具体场景）

```text
触发需求 -> 寻找入口 -> 执行操作 -> 等待反馈 -> 获得结果 -> 后续行为
```

**旅程还原方法：**
从评论中提取行为线索，将用户描述的操作片段还原成时序步骤。

每个旅程节点需标注：
- **用户行为**：用户在做什么
- **用户心理**：用户在想什么 / 期望什么
- **情感状态**：😊 满意 / 😐 中性 / 😟 不满 / 😡 愤怒
- **支撑证据**：来自哪条评论（引用原话）

### Stage 4：体验断点识别

分析旅程中哪些节点出现了问题，分三类：

**🔴 严重断点**：用户明确表示放弃、卸载、或强烈负面情绪  
**🟡 摩擦点**：用户感到困惑、绕路、需要多次尝试  
**🟢 满意点**：用户明确表示满意或超出预期

对每个断点，分析：
- 表层现象（用户说了什么）
- 深层原因（为什么会出现这个问题）
- 影响范围（有多少用户提到）

### Stage 5：设计建议

针对每个断点，给出可落地的设计建议。

建议格式：

```text
【断点】消息通知不及时
【核心诉求未满足】可靠性 + 安全感
【设计方向】
  1. 短期：优化系统通知权限引导，首次使用时明确告知用户如何开启通知
  2. 中期：增加“重要联系人”优先级设置，确保关键消息必达
  3. 长期：探索“消息送达确认”机制，给用户感知层面的安全感
【参考竞品做法】钉钉的“DING 消息”强提醒机制
```

建议优先级排序依据：
- 影响用户数量
- 情感强度
- 实现难度（需用户提供背景信息，否则只给方向不评估难度）

### Stage 6：输出报告

报告结构：

```text
# [产品名称] 用户研究报告
> 基于 XX 条公开用户评论 | 研究日期：XXXX

## 一、数据概览

## 二、核心诉求（Top 5）

## 三、用户旅程地图
[旅程图，用 Markdown 表格或文字图形呈现]

## 四、体验断点分析
### 🔴 严重断点
### 🟡 摩擦点
### 🟢 满意点

## 五、设计建议
[按优先级排列]

## 六、附录：代表性原始评论
```

---

## 注意事项

- **始终引用用户原话**作为证据支撑，不要只给结论
- **区分“用户说的”和“你的推断”**，推断要明确标注“推断：”
- 当数据不足以支撑某个结论时，明确说明“数据有限，建议补充收集”
- 语言跟随输入语言：输入中文则报告用中文，输入英文则报告用英文
- 对于敏感竞品对比，客观陈述，不做主观评价
